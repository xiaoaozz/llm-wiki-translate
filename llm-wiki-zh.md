# LLM 维基

一种使用 LLM 构建个人知识库的模式。

这是一份创意说明文档，设计用途是直接复制粘贴给你自己的 LLM 智能体（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传达整体思路，具体实现则由你的智能体与你协作完成。

## 核心理念

大多数人使用 LLM 处理文档的体验都类似于 RAG：你上传一批文件，LLM 在收到查询时检索相关片段，然后生成答案。这种方式确实有效，但每次提问时，LLM 都要从头重新发现知识。如果一个微妙的问题需要综合五份文档，LLM 每次都必须重新找到相关片段并把它们拼接起来。知识没有积累。NotebookLM、ChatGPT 文件上传功能以及大多数 RAG 系统都是这样运作的。

这里的思路有所不同。LLM 不只是在收到查询时从原始文档中检索信息，而是**逐步构建并维护一个持久化的维基**——一套位于你和原始资料之间、结构化且相互链接的 Markdown 文件。添加新资料时，LLM 不会仅仅为日后检索而建立索引。它会阅读资料、提取关键信息，并将其整合进现有维基：更新实体页面、修订主题摘要、标注新数据与旧论断相矛盾之处，并强化或质疑不断演进的综合结论。知识只需编译一次，之后便会*持续保持最新*，而不是每次查询时都重新推导。

这正是关键区别：**维基是一个持久存在、不断复利的成果。**交叉引用已经就位，矛盾已经标出，综合结论也已经涵盖了你读过的一切。你添加的每一份资料、提出的每一个问题，都会让维基变得更加丰富。

你从不（或很少）亲自编写维基——所有内容都由 LLM 编写和维护。你负责寻找资料、探索内容和提出正确的问题。LLM 则承担所有繁琐工作，包括总结、交叉引用、归档和记录，让知识库随着时间推移真正发挥作用。实际使用中，我会在一侧打开 LLM 智能体，在另一侧打开 Obsidian。LLM 根据我们的对话进行编辑，我则实时浏览结果——沿着链接探索、查看关系图谱、阅读更新后的页面。Obsidian 是 IDE，LLM 是程序员，而维基就是代码库。

这种模式可以应用于许多不同场景，例如：

- **个人管理**：追踪自己的目标、健康、心理和自我提升——归档日记、文章和播客笔记，逐步构建一幅结构化的自我画像。
- **研究**：用数周或数月深入钻研某个主题——阅读论文、文章和报告，并逐步构建一套全面的维基，其中的核心观点也会持续演进。
- **阅读一本书**：边读边归档每一章，为人物、主题、情节线索及其相互关系建立页面。读完时，你便拥有了一套内容丰富的伴读维基。可以想想 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝维基——数千个相互链接的页面，涵盖人物、地点、事件和语言，由一群志愿者历时多年共同建成。你也可以在阅读时为自己构建类似的维基，并让 LLM 完成所有交叉引用和维护工作。
- **企业/团队**：由 LLM 维护内部维基，以 Slack 讨论串、会议记录、项目文档和客户通话为信息来源，也可以让人参与审核更新。维基之所以能保持最新，是因为 LLM 承担了团队中没人愿意做的维护工作。
- **竞争分析、尽职调查、旅行规划、课程笔记、兴趣爱好深度研究**——任何需要长期积累知识，并希望将其有序组织而非任其散落的场景。

## 架构

整个系统分为三层：

**原始资料**——由你精选的源文档集合，包括文章、论文、图片和数据文件。它们不可变：LLM 只能读取，绝不修改。这是你的事实来源。

**维基**——一个存放 LLM 生成的 Markdown 文件的目录，包括摘要、实体页面、概念页面、比较、概览和综合分析。该层完全由 LLM 负责。它会创建页面，在新资料加入时更新页面，维护交叉引用，并确保所有内容一致。你负责阅读，LLM 负责写作。

**模式规范**——一份文档（例如 Claude Code 使用的 CLAUDE.md，或 Codex 使用的 AGENTS.md），用于告诉 LLM 维基采用怎样的结构、遵循哪些约定，以及在摄取资料、回答问题或维护维基时应执行哪些工作流程。这是关键的配置文件——正是它让 LLM 成为一个遵守规范的维基维护者，而不是泛用聊天机器人。随着你逐渐摸索出最适合自身领域的方法，你和 LLM 也会共同迭代这份规范。

## 操作

**摄取。**你将一份新资料放入原始资料集合，并让 LLM 处理它。一个示例流程是：LLM 阅读资料，与你讨论关键收获，在维基中编写摘要页面，更新索引，更新整个维基中相关的实体与概念页面，并在日志中追加一条记录。一份资料可能会影响 10～15 个维基页面。就我个人而言，我更喜欢每次摄取一份资料并全程参与——阅读摘要、检查更新，并指导 LLM 应重点关注什么。不过，你也可以一次批量摄取多份资料，只进行较少的监督。你可以自行制定适合自己风格的工作流程，并将其记录在模式规范中，供后续会话使用。

**查询。**你基于维基提出问题。LLM 会搜索相关页面、阅读内容，并综合生成带引用的答案。答案可以根据问题采用不同形式——Markdown 页面、对比表、幻灯片（Marp）、图表（matplotlib）或画布。关键洞见在于：**好的答案可以作为新页面归档回维基。**你请求的比较、某项分析或新发现的联系都很有价值，不应消失在聊天记录里。这样一来，你的探索成果就能像摄取的资料一样，在知识库中不断积累并产生复利。

**检查。**定期让 LLM 对维基进行健康检查。重点查找：页面之间的矛盾、已被新资料取代的过时论断、没有入站链接的孤立页面、提及过但尚无独立页面的重要概念、缺失的交叉引用，以及可通过网络搜索填补的数据空白。LLM 很擅长提出值得继续调查的新问题，以及建议寻找哪些新资料。随着维基不断增长，这有助于维持其健康状态。

## 索引与日志

随着维基不断扩展，两个特殊文件可以帮助 LLM（以及你）浏览维基。它们的用途各不相同：

**index.md** 以内容为导向。它是维基中所有内容的目录——列出每个页面的链接、一句话摘要，并可附加日期、资料数量等元数据。目录按照类别组织（实体、概念、资料等）。LLM 每次摄取资料时都会更新它。回答查询时，LLM 会先读取索引以找到相关页面，再深入阅读这些页面。这种方法在中等规模下（约 100 份资料、数百个页面）出奇地有效，而且无需搭建基于嵌入的 RAG 基础设施。

**log.md** 以时间为导向。它是一份仅追加的记录，记载发生过什么以及发生时间，包括摄取、查询和检查。一个实用技巧是：让每条记录都以统一前缀开头（例如 `## [2026-04-02] ingest | Article Title`），这样就能使用简单的 Unix 工具解析日志——`grep "^## \[" log.md | tail -5` 可以列出最近 5 条记录。日志展现了维基演变的时间线，也能帮助 LLM 了解最近完成了哪些工作。

## 可选：CLI 工具

在某个阶段，你可能会想开发一些小工具，帮助 LLM 更高效地操作维基。最显而易见的是针对维基页面的搜索引擎——规模较小时，索引文件已经足够；但随着维基增长，你会需要真正的搜索功能。[qmd](https://github.com/tobi/qmd) 是个不错的选择：它是一个面向 Markdown 文件的本地搜索引擎，支持混合 BM25/向量搜索和 LLM 重排序，而且完全在设备端运行。它既提供 CLI（LLM 可以通过 shell 调用），也提供 MCP 服务器（LLM 可以将其作为原生工具使用）。你也可以自己构建更简单的工具——等到确有需要时，让 LLM 帮你凭感觉快速写一个朴素的搜索脚本即可。

## 技巧与窍门

- **Obsidian Web Clipper** 是一款能将网页文章转换为 Markdown 的浏览器扩展。它非常适合用来快速把资料加入原始资料集合。
- **将图片下载到本地。**在 Obsidian 的“设置”→“文件与链接”中，将“附件默认存放路径”设为一个固定目录（例如 `raw/assets/`）。然后在“设置”→“快捷键”中搜索“下载”，找到“下载当前文件的附件”，并为其绑定快捷键（例如 Ctrl+Shift+D）。剪藏一篇文章后，按下快捷键，所有图片就会下载到本地磁盘。这一步并非必需，但很实用——LLM 可以直接查看和引用图片，而不必依赖可能失效的 URL。需要注意的是，LLM 无法在一次处理中原生读取带有内嵌图片的 Markdown。解决方法是先让 LLM 阅读文本，再单独查看部分或全部引用的图片，以获取额外上下文。操作稍显笨拙，但已经足够好用。
- **Obsidian 的关系图谱**是观察维基整体形态的最佳方式——哪些内容相互连接、哪些页面是枢纽、哪些页面处于孤立状态。
- **Marp** 是一种基于 Markdown 的幻灯片格式，Obsidian 也有对应插件。它适合直接使用维基内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可以对页面的 frontmatter 执行查询。如果你的 LLM 会在维基页面中添加 YAML frontmatter（标签、日期、资料数量），Dataview 就可以生成动态表格和列表。
- 维基本质上只是一个由 Markdown 文件组成的 Git 仓库。你天然就能获得版本历史、分支和协作能力。

## 为什么这种方式有效

维护知识库最繁琐的部分不是阅读或思考，而是记录和整理：更新交叉引用、确保摘要及时更新、标注新数据与旧论断冲突之处，以及维持数十个页面之间的一致性。人们之所以放弃维基，是因为维护负担的增长速度超过了它所带来的价值。LLM 不会感到无聊，不会忘记更新交叉引用，还能在一次操作中处理 15 个文件。由于维护成本趋近于零，维基便能持续得到维护。

人的工作是精选资料、指导分析、提出好问题，并思考这一切意味着什么。除此之外的所有工作都交给 LLM。

这一理念在精神上与万尼瓦尔·布什（Vannevar Bush）于 1945 年提出的 Memex 有关——一个私人的、经过精选的知识存储系统，文档之间通过联想路径相连。相比后来实际形成的万维网，布什的构想与这里的理念更为接近：它是私有的、经过主动整理的，而且文档之间的联系与文档本身同样宝贵。他当时无法解决的问题是：谁来负责维护？现在，LLM 可以承担这项工作。


## 说明

本文档有意保持抽象。它描述的是一种理念，而非某种具体实现。确切的目录结构、模式规范约定、页面格式和工具链，都取决于你的领域、偏好以及所选的 LLM。上文提到的一切都是可选且模块化的——选择有用的部分，忽略不适用的部分。例如：你的资料可能只有文本，因此完全不需要处理图片；你的维基可能规模很小，只使用索引文件就已足够，无需搜索引擎；你可能不关心幻灯片，只想要 Markdown 页面；你也可能希望采用一套完全不同的输出格式。正确的使用方式是将本文档分享给你的 LLM 智能体，然后与它共同构建一个符合你需求的版本。本文档唯一的任务就是传达这种模式，其余部分交给你的 LLM 即可。
